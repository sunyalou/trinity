# Feature: Agent Guardrails (GUARD-001 / 002 / 003)

## Overview

Guardrails are a **defense-in-depth safety layer** baked into every Trinity
agent container. They combine three independent controls, all enforced
*inside* the agent (not the backend):

1. **Bash command deny-list** — regex patterns block dangerous shell commands
   (`rm -rf /`, force-push, pipe-to-shell, fork bombs, fs format, host
   shutdown, …) via a Claude Code `PreToolUse` hook on the `Bash` tool.
2. **File path deny-list** — glob patterns block writes/edits to credential
   files, SSH/AWS/cloud config, and Trinity's own protected directories via a
   `PreToolUse` hook on `Write|Edit|NotebookEdit|MultiEdit`.
3. **CLI budget & scope controls** — turn caps (`max_turns_chat` /
   `max_turns_task`), a wall-clock `execution_timeout_sec`, and a
   `disallowed_tools` list passed straight to the `claude` CLI on every
   invocation.

A **platform baseline** (`guardrails-baseline.json`, baked into the base image,
root-owned `0444`) always applies and is **not** overridable by the agent or by
owners. Owners may layer **narrow per-agent overrides** — numeric caps and
literal substring deny-lists only; never regex, never the credential scanner —
via `PUT /api/agents/{name}/guardrails`. A `PostToolUse` scanner additionally
logs (never blocks) credential-shaped strings in `Bash` output.

The whole mechanism is **fail-closed**: any hook error exits `2` (deny), the
runtime config is generated by root at boot so the agent can't rewrite it, and
the hooks themselves live in `/opt/trinity/hooks/` (root-owned, path-protected).

## User Story

As an agent owner, I want a dangerous-command and dangerous-write safety net
that's always on, plus the ability to tighten turn/timeout budgets and add my
own deny rules for a specific agent — without being able to *weaken* the
platform's baseline protections.

## Entry Points

- **UI (Agent Detail)**: owner-only **Settings** tab (renamed from "Guardrails", #1108/#1122) — `GuardrailsPanel.vue` rendered unchanged as **section #1** inside `components/settings/SettingsPanel.vue`, wired in `AgentDetail.vue:187` (tab pushed at `647`, gated on `agent.can_share`). Old `?tab=guardrails` deep links resolve via `TAB_ALIASES` (`guardrails → settings`). Exposes `max_turns_chat` / `max_turns_task` only (#967 / #992).
- **API (read)**: `GET /api/agents/{name}/guardrails` — any authenticated user; 404 if no container.
- **API (write)**: `PUT /api/agents/{name}/guardrails` — **owner-only**; empty body clears overrides. Requires agent restart to apply.
- **Agent creation**: overrides serialized into the `AGENT_GUARDRAILS` env var (`crud.py`).
- **Agent restart**: env var re-serialized from DB (`lifecycle.py`); runtime config regenerated at boot (`startup.sh` → `write-runtime-config.py`).
- **Tool-call time**: Claude Code `PreToolUse` / `PostToolUse` hooks fire on every `Bash` / file-write tool call.

> **UI scope is narrower than the API.** The Settings tab's Guardrails section only edits the two turn caps; `execution_timeout_sec` and the `extra_bash_deny` / `extra_path_deny` / `disallowed_tools` lists remain API-only and are *preserved* by the panel's whole-config PUT (see Frontend Layer).

Follows the standard three-layer pattern (Invariant #1).

**Validation contract** (intentionally narrow — owners cannot extend regex or the credential scanner):

| Field | Type | Bound |
|-------|------|-------|
| `max_turns_chat` | int | 1–500 |
| `max_turns_task` | int | 1–500 |
| `execution_timeout_sec` | int | 60–7200 |
| `extra_bash_deny` | list[str] | ≤50 items, each 1–256 chars |
| `extra_path_deny` | list[str] | ≤50 items, each 1–256 chars |
| `disallowed_tools` | list[str] | ≤50 items, each 1–256 chars |

`bool` is explicitly rejected for numeric fields (`isinstance(value, bool)` guard at 582). Body is a raw `dict` (not a centralized Pydantic model) — the validator *is* the contract.

## Agent-Side Enforcement (`docker/base-image/`)

This is where guardrails actually *bite*. All hooks are root-owned in `/opt/trinity/hooks/` and registered permanently in the base image's `~/.claude/settings.json` — the agent cannot deregister them (the settings file is itself in `path_deny`).

### Hook registration — `hooks/claude-settings.json`

| Event | Matcher | Hook(s) |
|-------|---------|---------|
| `PreToolUse` | `Bash` | `bash-guardrail.py` |
| `PreToolUse` | `Edit\|Write\|NotebookEdit\|MultiEdit` | `file-guardrail.py`, then `read-only-guard.py` |
| `PostToolUse` | `Bash` | `output-scanner.py` |

### Shared library — `hooks/lib.py` (103 lines)

| Symbol | Behavior |
|--------|----------|
| `load_config()` (39-53) | runtime → baseline → `{}`; never raises |
| `read_stdin_json()` (56-63) | parse hook stdin; **fail-closed** (exit 2) on bad JSON |
| `compile_patterns()` (66-78) | compile regex list; skips + logs bad patterns |
| `deny(msg, **fields)` (81-85) | log to `/logs/guardrails.jsonl`, stderr, `exit(2)` |
| `allow()` (88-90) | `exit(0)` |
| `run_hook(main_fn)` (93-102) | wraps `main()`; any uncaught exception → log + `exit(2)` (**fail-closed**) |
| `log_event()` (28-36) | append structured JSONL; never raises |

**Hook protocol**: exit 0 (silent) = allow; exit 2 + stderr = deny (message shown to model); exit 1 / uncaught = Claude treats as non-blocking, so hooks deliberately convert all errors to exit 2.

### Bash deny — `hooks/bash-guardrail.py` (54 lines)

Loads config → compiles `bash_deny` (baseline regex) → first match wins → `deny`. Then checks `extra_bash_deny` (per-agent literal substrings). `allow()` if clean. Logs `tool`, `pattern`, `command_prefix[:120]` on deny.

### File deny — `hooks/file-guardrail.py` (77 lines)

`_normalise(path)` → absolute (expands `~`, prefixes `/home/developer` for relative). `_matches_glob` runs `fnmatch` against **both** the full path and the basename, so `.env` patterns catch nested copies. Checks `path_deny` (baseline globs) then `extra_path_deny` (literals). `deny` with `tool`, `path`, `pattern`.

### Credential scanner — `hooks/output-scanner.py` (58 lines, PostToolUse)

Scans combined `stdout`/`stderr`/`output` of Bash against `credential_patterns`. Logs **matched pattern names only** (never the value) to `guardrails.jsonl`. **Always exits 0** — log-only, never blocks.

### Runtime config merge — `hooks/write-runtime-config.py` (83 lines)

Runs as **root** via `sudo` from `startup.sh`. Reads baseline + `$AGENT_GUARDRAILS`, sanitizes overrides (`_sanitise_number` clamps to bounds, `_sanitise_string_list` caps 50×256), and atomically writes `/opt/trinity/guardrails-runtime.json` via tmp-file + `chmod 0444` + `os.replace`. Only `OVERRIDABLE_NUMBERS` (3) + `OVERRIDABLE_STRING_LISTS` (3) are merged; regex lists and `credential_patterns` are never touched. Malformed `AGENT_GUARDRAILS` → ignored (baseline stands).

### CLI budget/scope — `agent_server/services/_runtime_config.py` (33 lines) + consumers

`_load_guardrails()` (runtime → baseline → `{}`) is read on **every** Claude invocation, so backend-driven changes (via container recreation) apply without restarting the agent-server process. Defaults: `_DEFAULT_MAX_TURNS_CHAT=50`, `_DEFAULT_MAX_TURNS_TASK=50`, `_DEFAULT_EXECUTION_TIMEOUT_SEC=1800` (30 min).

| Consumer | Lines | Applies |
|----------|-------|---------|
| `claude_code.py` (chat) | 209-222 | `--max-turns max_turns_chat`, `--disallowedTools`, wall-clock `execution_timeout_sec` |
| `headless_executor.py` (task) | 323-344, 647 | `--disallowedTools`, `--max-turns max_turns_task`; turn-limit error suggests raising `max_turns_task` |

## Configuration & Defaults

### Baseline — `docker/base-image/hooks/guardrails-baseline.json`

Immutable, root-owned `0444`. Contents:

- **`bash_deny`** (10 regex+reason rules): recursive root/home delete, `chmod 777`, pipe-to-shell, git force-push, `kill -9 1`, raw `dd of=/dev/sd…`, `mkfs.*`, fork bomb, read-only-config tamper, host shutdown/reboot/poweroff.
- **`path_deny`** (12 globs): `.env`, `.env.*`, `.mcp.json`, `.credentials.enc`, `~/.ssh/*`, `~/.aws/*`, `~/.gcp/*`, `~/.claude/settings.json`, `~/.claude/settings.local.json`, `~/.trinity/read-only-config.json`, `/opt/trinity/*`, `/etc/claude-code/*`.
- **`credential_patterns`** (9): Anthropic key/OAuth, OpenAI, GitHub PAT (classic + fine-grained), AWS access key, Slack bot/user token, Google API key.
- **Budgets**: `max_turns_chat: 50`, `max_turns_task: 50`, `execution_timeout_sec: 1800`.
- **Override slots** (empty by default): `extra_bash_deny: []`, `extra_path_deny: []`, `disallowed_tools: []`.

## Relationship to Read-Only Mode (CFG-007 / #887)

Guardrails and read-only mode are **independent, complementary** layers:

- The read-only feature's config file `~/.trinity/read-only-config.json` is itself **protected by guardrails** — both by `path_deny` (block Write/Edit) *and* by a dedicated `bash_deny` regex (block shell redirects/pipes at it). Defense-in-depth: an agent can't disable read-only mode by overwriting its own config.
- `file-guardrail.py` and `read-only-guard.py` both run on the same `Edit|Write|…` matcher. Guardrails fire **regardless of read-only state** — disabling read-only does *not* re-open `path_deny` targets.
- `#887` moved the read-only guard into the base image (`/opt/trinity/hooks/`, root-owned) and made it fail-closed via the same `run_hook()` wrapper documented above.

See [read-only-mode.md](read-only-mode.md).

## Security Properties

1. **Baseline immutable** — image-baked, root-owned `0444`; agent cannot read-modify-write it.
2. **Runtime config root-generated** — written by `sudo` at boot, `0444`; agent process cannot rewrite it post-start.
3. **Hooks tamper-proof** — in `/opt/trinity/*` (path-denied) and registered in `~/.claude/settings.json` (also path-denied).
4. **Override scope minimal** — numeric caps + literal substrings only; regex and credential scanner are platform-owned.
5. **Fail-closed everywhere** — bad stdin, bad config, or any uncaught hook exception → deny (exit 2).
6. **No value leakage** — credential scanner logs pattern *names*, never matched secrets.
