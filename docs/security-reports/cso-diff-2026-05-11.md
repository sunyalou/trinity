## Finding #3 — Cold-turn session lock removed (race regression)

- **Severity:** MEDIUM
- **Confidence:** 9/10
- **Status:** VERIFIED
- **Phase:** 9 — A04 Insecure Design (race condition / data integrity)
- **Category:** Platform Patterns
- **File:** `src/backend/routers/sessions.py:296-373`

**Description:** This branch removes the per-session cold-turn lock added by PR #779 (commit `51f7791c`, "fix(session-tab): lock cold-turn POSTs on a per-session key"). The reverted code held `session_lock:cold:{session_id}` around the cold-path `claude --print` invocation. Removing it returns to the pre-#779 race: two concurrent first-turn POSTs on the same `agent_sessions` row both start fresh Claude sessions, both write `cached_claude_session_id`, one wins the DB update, and the loser's JSONL is orphaned until the 6h+1h periodic sweep.

**Exploit scenario:** Not an authenticated-attacker exploit — a single user double-clicking "send" on a fresh session triggers it. Impact: (1) duplicate cost — user is billed for two Claude invocations on the first turn, (2) orphaned JSONL stays on disk for up to 7h before reaping, (3) `consecutive_resume_failures` may increment incorrectly on the loser's row.

**Recommendation:** Rebase onto `origin/dev` to inherit `51f7791c`. If the diff intentionally restructures the lock primitive, the cold path still needs serialization on `session_id` — restore the `session_lock:cold:{session_id}` key with the same TTL semantics.
