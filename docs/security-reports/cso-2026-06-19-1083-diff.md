## Context

The branch implements **fire-and-forget dispatch** (#1083): eligible autonomous turns
(`{schedule, webhook}`) are dispatched to the agent with a **202 ACK** and finalized via a new
**result-callback endpoint** (`POST /api/agents/{name}/executions/{id}/result`), so a wedged
turn holds zero backend coroutine/slot beyond its lease. New security-relevant surfaces: an
inbound HTTP callback endpoint, agent-side HTTP delivery authenticated with the agent's own
MCP key, new config flags (`DISPATCH_ASYNC`), and a refactor of the terminal applier
(`apply_result`).

### LOW — Resolved in this branch

1. **(Injection / defense-in-depth) Agent-side `execution_id` used unsanitized in a filesystem path — RESOLVED.**
   `result_callback._pending_path()` builds `~/.trinity/pending-results/{execution_id}.json`
   and the callback URL from the backend-supplied `execution_id`. The value is backend-generated
   (`secrets.token_urlsafe(16)` / UUID) and not attacker-controlled in normal operation, but a
   path-traversal value (`../…`) would have escaped the pending dir if the backend were ever
   compromised or the id format changed.
   **Fix (commit on this branch):** added a strict allowlist guard
   (`_SAFE_EXECUTION_ID = ^[A-Za-z0-9_-]{1,128}$` + `_is_safe_execution_id()`) enforced at the
   async-dispatch entry point `try_spawn_async()` — an `execution_id` outside the urlsafe-token /
   UUID charset now falls back to **synchronous** handling and never reaches the path build or
   the callback URL. The resend path is bounded by a bare `*.json` glob stem. Covered by
   parametrized unit tests in `tests/unit/test_1083_result_callback.py`
   (`test_unsafe_execution_id_returns_false`, `test_is_safe_execution_id_accepts_real_ids`).

### Informational (pre-existing, not introduced by this diff)

- **Terminal `error` string persisted without backend-side `sanitize_text`.** `apply_result`
  writes `envelope.error` to the `error` column / activity without re-running the credential
  sanitizer (`metadata` is `sanitize_dict`'d; `response`/`execution_log` are sanitized). This
  relies on agent-side sanitization and is **unchanged** from `origin/dev` — the `apply_result`
  refactor faithfully preserves it across both the sync path and the new callback path
  (`_envelope_from_http_exception` → `str(detail)[:500]`). Track as a future platform hardening
  item; out of scope for this PR.
- **`except BaseException` in `_run_and_report` (agent side)** converts any `BaseException`
  (incl. `CancelledError`) to a failed envelope rather than propagating cancellation.
  Robustness/style, not security; `noqa`'d with justification.
- **Agent-server `/api/task` remains unauthenticated** (network-isolation reliant). Pre-existing;
  the `async_result` passthrough does not widen it — the callback requires the victim agent's
  own env-injected MCP key plus the backend ownership gate, so it cannot be abused cross-tenant.

## Verified clean

- **Secrets**: none in the diff. The only new env var is `DISPATCH_ASYNC` (boolean, default
  `false`).
- **Dependencies**: no new packages; `result_callback.py` uses `httpx` (already a dependency).
- **Auth boundary — callback endpoint** `POST /api/agents/{name}/executions/{id}/result` is
  fail-closed:
  - `Bearer` required → 403; validated via `db.validate_mcp_api_key(track_usage=False)` +
    `heartbeat_service.authorize_heartbeat(res, agent_name)` — an agent may finalize **only its
    own** executions (user/system/null/wrong-agent keys → 403).
  - Ownership mismatch → **404 (not 403)**: no execution-ID existence leak / enumeration.
  - Durable **async-marker gate** (`claude_session_id == 'dispatched_async'`) → 409 otherwise:
    a sync/interactive row the backend is mid-await on can never be terminal-written here.
  - Idempotent replay for authoritative terminals; FAILED deliberately falls through so a late
    SUCCESS corrects a reaper `LEASE_EXPIRED` via the CAS (Codex #2).
  - Body-size caps → clean **413** (4M-char response / 16MB log).
- **Cross-tenant abuse**: a malicious agent can only affect its **own** executions/breaker
  (status/error_code are self-scoped).
- **Injection**: all new SQL is SQLAlchemy-parameterized (`select`/`update`, `.in_()` tuples;
  f-strings only build bound *values*). No `subprocess`/`eval`/`os.system`. No SSRF — the
  callback URL is built from the trusted `TRINITY_BACKEND_URL` env + the agent's own name; a
  poisoned pending-results file cannot impersonate another agent (backend `authorize_heartbeat`
  rejects mismatch → permanent 403 → dropped). With the new guard, `execution_id` can no longer
  inject into the path or URL.
- **Credential handling**: `TRINITY_MCP_API_KEY` read from env, sent as `Bearer`, never logged.
- **Capacity-bypass hardening (positive)**: `slot_service.release_slot` now gates the
  BACKLOG-001 drain on `removed`, and `apply_result` gates slot release on the CAS win — a
  replayed/late callback cannot over-admit past `max_parallel_tasks`.
- **Configuration**: `DISPATCH_ASYNC` default `false`, mirrored in **both** `docker-compose.yml`
  and `docker-compose.prod.yml` (avoids the #1039 packaging-gap class).
